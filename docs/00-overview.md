# 00. 프로젝트 개요

## 왜 이 프로젝트를 만들었는가

마이크로서비스 아키텍처에서 서비스 간 통신은 크게 두 가지로 나뉜다:

1. **동기 통신** — REST API, gRPC 등으로 직접 호출
2. **비동기 통신** — 메시지 브로커(Kafka, RabbitMQ 등)를 통한 이벤트 기반 통신

동기 통신은 구현이 간단하지만, 서비스 간 **강한 결합**이 생긴다. A 서비스가 B 서비스를 직접 호출하면, B가 죽었을 때 A도 영향을 받는다.

비동기 통신은 서비스 간 **느슨한 결합**을 만든다. A는 메시지를 Kafka에 던지고 끝. B가 언제 처리하든 A는 상관없다. 이것이 이벤트 기반 아키텍처(EDA)의 핵심이다.

이 프로젝트는 **Spring Cloud + Apache Kafka** 조합으로 EDA를 직접 구현하고 체험하기 위해 만들었다.

## 무엇을 학습할 수 있는가

### Spring Cloud

- **Eureka** — 서비스 디스커버리: 각 서비스가 자기 위치를 등록하고, 다른 서비스를 찾는 메커니즘
- **Spring Cloud Gateway** — API 게이트웨이: 클라이언트의 단일 진입점, 라우팅, 로드밸런싱

### Apache Kafka

- **Producer/Consumer** — 메시지를 발행하고 구독하는 기본 패턴
- **Topic** — 메시지가 저장되는 논리적 채널
- **Consumer Group** — 같은 토픽을 여러 서비스가 독립적으로 구독하는 방법
- **JSON 직렬화** — Java 객체를 JSON으로 변환하여 Kafka에 전송하는 방법
- **크로스 서비스 타입 매핑** — 서로 다른 패키지의 클래스 간 역직렬화 문제 해결

### 인프라

- **Docker Compose** — 다중 컨테이너 오케스트레이션
- **멀티스테이지 빌드** — Docker 안에서 Gradle 빌드 후 경량 JRE 이미지로 실행
- **Healthcheck + depends_on** — 컨테이너 기동 순서 제어

## 도메인: 온라인 주문 시스템

실제 비즈니스와 유사한 시나리오를 사용하여 학습 효과를 높였다:

```
고객이 주문을 넣으면
  → 결제가 자동으로 처리되고
    → 결과 알림이 발송된다
```

이 흐름에서 각 단계는 **독립된 마이크로서비스**이며, **Kafka 이벤트**로 연결된다.

| 서비스 | 역할 | 비유 |
|--------|------|------|
| Order Service | 주문 접수 | 카운터 직원 |
| Payment Service | 결제 처리 | PG사 |
| Notification Service | 알림 발송 | 문자/이메일 서비스 |
